Objective
Build a product catalog application three times using Next.js to implement and compare Client-Side Rendering (CSR), Server-Side Rendering (SSR), and Static Site Generation (SSG). You will learn to use performance analysis tools like Lighthouse and Chrome DevTools to measure Core Web Vitals, understand the trade-offs of each rendering strategy, and make informed architectural decisions. This is a crucial skill for building high-performance, SEO-friendly web applications.

Description
1. Project Background and Core Concepts
Welcome to a deep dive into the world of web rendering strategies. In modern web development, how and when your application's HTML is generated has a profound impact on user experience, performance, and search engine optimization (SEO). This project is designed to give you hands-on experience with the three dominant rendering patterns—Client-Side Rendering (CSR), Server-Side Rendering (SSR), and Static Site Generation (SSG)—by building the same application three different ways and measuring the outcomes with professional-grade tools. You will not only learn the theory but see the concrete effects on performance metrics like Time to First Byte (TTFB) and Largest Contentful Paint (LCP).

What are Rendering Strategies?
At its core, rendering is the process of turning your application's code (e.g., React components) into the HTML, CSS, and JavaScript that a browser can understand and display to a user. The key distinction between strategies is where this conversion happens (on the client's browser or on the server) and when it happens (at build time, at request time, or after the page loads in the browser).

Client-Side Rendering (CSR): This was the default for Single-Page Applications (SPAs) for many years. The server sends a minimal HTML file, often just a single <div id="root"></div> and a link to a large JavaScript bundle. The user's browser then downloads and executes this JavaScript, which in turn makes API calls to fetch data and renders the components into the DOM.

Pros: Rich, interactive experiences; no per-request server computation load after the initial static files are served.
Cons: Slow initial load time (high Time to First Contentful Paint - FCP) as nothing is visible until the JS bundle is downloaded, parsed, and executed. Poor for SEO, as web crawlers may only see the empty HTML shell. This is often called the "blank white screen of death."
Server-Side Rendering (SSR): With SSR, every time a user requests a page, the server fetches the necessary data, renders the React components into a full HTML string, and sends that complete HTML document to the browser. The browser can immediately display the content, resulting in a very fast FCP and LCP. However, the page is not yet interactive. The browser must still download the JavaScript bundle and execute it in a process called "hydration," which attaches event listeners to the server-generated DOM.

Pros: Excellent for SEO as crawlers get fully-rendered HTML. Fast perceived performance because users see content almost instantly.
Cons: Higher Time to First Byte (TTFB) because the server has to do work on every request. The server is a bottleneck and must be scaled to handle traffic. There's a period known as the "hydration window" where the page looks interactive but isn't, which can frustrate users.
Static Site Generation (SSG): SSG takes rendering a step further by performing it at build time. When you deploy your application, a build process runs, fetches data, and pre-renders every page into a static HTML file. These files are then deployed to a Content Delivery Network (CDN). When a user requests a page, the CDN serves the pre-built HTML from a server geographically close to them.

Pros: The fastest possible performance. Extremely low TTFB, excellent reliability, and security. Perfect for SEO.
Cons: The content can become stale. If the data changes, you must trigger a new build to update the site. This limitation is mitigated by Incremental Static Regeneration (ISR), a Next.js feature that allows pages to be re-generated in the background after a certain time has passed, blending the speed of static with the freshness of dynamic rendering.
Core Web Vitals Explained
Google's Core Web Vitals are a set of standardized metrics to help developers understand the real-world user experience of a page. You will be measuring these extensively.

Largest Contentful Paint (LCP): Measures loading performance. It marks the point when the page's main content has likely loaded. A good LCP is 2.5 seconds or less.
First Input Delay (FID) / Total Blocking Time (TBT): Measures interactivity. FID measures the time from when a user first interacts with a page (e.g., clicks a button) to the time when the browser is actually able to respond. TBT is a lab metric that measures the total amount of time the main thread was blocked, preventing user input. A good TBT is under 300ms.
Cumulative Layout Shift (CLS): Measures visual stability. It quantifies how much unexpected layout shifts occur during the lifespan of the page. A good CLS is 0.1 or less.
Time to First Byte (TTFB): A foundational metric that measures the time between the request for a resource and when the first byte of a response begins to arrive. It's a key indicator of server responsiveness.

3. Implementation Guide: Building the Product Catalog
You will build a product catalog application with two main pages:

Product List Page: Displays a grid of 20 products.
Product Detail Page: A dynamic page showing details for a single product.
The data will be fetched from the free public API: https://dummyjson.com/products. For consistency in your tests, you might consider saving the output of a call to ?limit=20 into a local JSON file to ensure the data doesn't change during your benchmarking.

Step 3.1: Shared UI Components
First, create the reusable UI components that will be shared across all three versions. Place them in a components/ directory.

ProductCard.js: A component to display a product's image, title, and price.
ProductGrid.js: A component that takes an array of products and maps over them, rendering a ProductCard for each one in a responsive grid.
Header.js: A simple header with the application title and a cart status indicator (e.g., "Cart: 0").
Layout.js: A wrapper component that includes the Header and renders the page content passed as children.
Step 3.2: Version 1 - Client-Side Rendering (CSR)
This version mimics a classic React SPA. The server's only job is to provide the initial JavaScript.

Product List Page (pages/products/index.js):
Use the useState hook to manage products and loading states.
Use the useEffect hook to fetch data from the API when the component mounts.
While loading is true, display a skeleton UI (e.g., gray placeholder boxes) to improve perceived performance. This is crucial for a good user experience with CSR.
Once the data arrives, update the state and render the ProductGrid component.
Product Detail Page (pages/products/[id].js):
Use the useRouter hook from next/router to get the product ID from the URL.
Use useEffect to fetch data for that specific product ID (e.g., https://dummyjson.com/products/1).
There will be no getServerSideProps or getStaticProps in this version. Next.js will automatically treat these pages as CSR.

Step 3.3: Version 2 - Server-Side Rendering (SSR)
Here, the server does the heavy lifting for every request.

Product List Page (pages/products/index.js):
Export an async function getServerSideProps().
Inside this function, fetch the list of 20 products.
Return the fetched products as props: return { props: { products } }.
The page component will receive products as a prop and can pass it directly to the ProductGrid.
Product Detail Page (pages/products/[id].js):
The getServerSideProps function receives a context object. You can get the product ID from context.params.id.
Fetch the data for that specific product and return it as props.
Step 3.4: Version 3 - Static Site Generation (SSG)
This version aims for maximum speed by pre-rendering everything at build time.

Product List Page (pages/products/index.js):
Export an async function getStaticProps().
Fetch the product list inside this function.
Return the products as props. For this task, also add revalidate: 60 to enable Incremental Static Regeneration (ISR). This tells Next.js to re-generate the page in the background if a request comes in after 60 seconds, ensuring content is never too stale.
Product Detail Page (pages/products/[id].js):
This is a dynamic route, so you need both getStaticProps and getStaticPaths.
getStaticPaths: This function tells Next.js which product IDs to pre-render at build time. Fetch all 20 products and return their IDs in the required format: return { paths: [ { params: { id: '1' } }, ... ], fallback: 'blocking' }.
fallback: false would mean any product ID not listed in paths results in a 404.
fallback: 'blocking' is a great hybrid: if a user requests a page not yet generated, Next.js will SSR it on the first request and then cache the static HTML for all subsequent requests. This is perfect for large e-commerce sites with thousands of products.
getStaticProps: This function will receive the id from params and fetch the data for that specific product to render the page.
4. Benchmarking Protocol
This is the most critical phase. You will deploy each of the three versions to Vercel (which has a generous free tier) and perform a rigorous, repeatable set of performance tests. Do not test on localhost, as it eliminates network latency and will give you misleadingly optimistic results.

Step 4.1: Deploy to Vercel
Create three separate projects on Vercel, one for each of your rendering strategies. This will give you three distinct URLs to test.

Step 4.2: Install and Run Lighthouse CLI
Automated testing is key to reliable metrics. Install the Lighthouse CLI globally:

npm install -g lighthouse
Create a results/ folder in your project. Run the audit for each deployment's product list page. It is essential to run each command multiple times (at least 3) and take the median score, as network conditions can cause variability.

# Example for CSR Desktop
lighthouse https://your-csr-app.vercel.app/products \
  --output json --output-path ./results/csr-desktop.json \
  --preset=desktop --throttling-method=devtools \
  --chrome-flags="--headless"

# Example for CSR Mobile
lighthouse https://your-csr-app.vercel.app/products \
  --output json --output-path ./results/csr-mobile.json \
  --chrome-flags="--headless"
Repeat this for SSR and SSG URLs, for both desktop and mobile presets.

Step 4.3: Parse Lighthouse Results
Create a Node.js script (parse-results.js) to extract the key metrics from the generated JSON files. This allows you to easily compare the numbers side-by-side.

// parse-results.js
const fs = require('fs');
const path = require('path');

function parseReport(filePath) {
  const report = JSON.parse(fs.readFileSync(filePath, 'utf-8'));
  const audits = report.audits;
  return {
    TTFB: audits['server-response-time'].numericValue.toFixed(2),
    FCP: audits['first-contentful-paint'].numericValue.toFixed(2),
    LCP: audits['largest-contentful-paint'].numericValue.toFixed(2),
    TTI: audits['interactive'].numericValue.toFixed(2),
    TBT: audits['total-blocking-time'].numericValue.toFixed(2),
    CLS: audits['cumulative-layout-shift'].numericValue.toFixed(3),
    Score: report.categories.performance.score * 100,
  };
}

Step 4.4: Manual DevTools Analysis
Automated tools are great, but manual inspection reveals nuance.

Network Waterfall: Open each deployed site in Chrome. Go to DevTools -> Network tab. Set throttling to "Slow 3G". Reload the page. Observe the waterfall chart.
For CSR, you'll see a tiny HTML file followed by a large JS file, and then API requests. Notice the long gap before anything renders.
For SSR, you'll see a longer TTFB for the initial HTML document, but it will contain all the content.
For SSG, the TTFB for the initial document should be extremely short.
Performance Trace: Go to the Performance tab. Start recording a trace and then reload the page. Stop recording after the page is fully interactive. Look for the LCP marker and the long, red-striped "Long Tasks" that contribute to TBT. This is where you can visually identify the SSR hydration window—the time between LCP and when the main thread is free.
Take screenshots of your most illustrative waterfall and performance traces for your final analysis.

6. FAQ
Q: Why do my Lighthouse scores change on every run?
A: Network conditions, server load, and even background processes on your own machine can introduce variability. This is why running tests multiple times and taking the median is a professional best practice. Using the --throttling-method=devtools flag helps standardize the network simulation.
Q: Can I just test on http://localhost:3000?
A: You should not rely on localhost for your final benchmarks. It completely bypasses real-world network latency. An SSR page might seem instant on localhost because the "server" is on the same machine, but its TTFB will be much higher once deployed. Your SSG and CSR pages will also appear artificially fast. Always test on a public deployment.
Q: For SSG, what's the difference between fallback: 'blocking' and fallback: true?
A: Both handle pages not generated at build time. 'blocking' will make the user wait for the server to SSR the page, so they get the full content in the initial response (better for SEO). true will immediately send a static shell of the page (with props being empty), and the client-side will fetch the data and populate the page, similar to CSR. 'blocking' provides a better user and crawler experience in most cases.
Q: How does ISR revalidate actually work?
A: When a request is made to a page after the revalidate window has passed, Vercel serves the stale (cached) page to the user immediately. Simultaneously, in the background, it re-renders the page with fresh data. Once the new page is ready, it silently replaces the stale page in the cache. The next user to request the page will get the newly generated version. This ensures users always get a fast response while data is updated gracefully.
